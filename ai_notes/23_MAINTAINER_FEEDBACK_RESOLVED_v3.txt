S1 — Word slot ID — ACCEPTED, CLARIFIED
--------------------------------------------------------------------------------
DECISION: Implement slot IDs in Phase 1 instrumentation.
CLARIFY:
  - Gentle autocorrect applies when user SWITCHES from swipe to tap at the SAME
    slot (same word position) after a bad swipe — even if they edit again.
  - If user continues swiping (no tap retry), autocorrect behaves normally.
  - Slots must support journal back several paragraphs (slider window).

--------------------------------------------------------------------------------
S2 — Confidence / commit — REVISED (major)
--------------------------------------------------------------------------------
REJECTED: Using near-tie whole-word scores to withhold commit or auto-pick #2
  (e.g. food → find via "second best"). That caps accuracy and mimics broken
  commercial keyboards.

ACCEPTED:
  - Per-segment, per-KEY confidence during swipe (dictionary narrows letter by
    letter). Whole-word confidence is emergent from path, not guessed upfront.
  - On finger lift: commit the top dictionary match to the text field.
  - Runner-up words stay in suggestion strip (not withheld) for user correction.
  - Heatmap + correction history bias keys/paths so top match becomes correct
    over time (e.g. F→gap I/O diagonal UP-RIGHT trains toward I not O).
  - Row constraint: stroke entering bottom row must admit bottom-row letters in
    path (anti "food" when user hit N — geometry + narrowing, not LM guess).

NEW CANONICAL EXAMPLE: NEXT swipe (see 20_FEATURE_*_v3.txt appendix).

S2 old "confidence gate" doc language is retired; strip learning replaces it.

P3 — Weight decay — ACCEPTED WITH NUANCE
--------------------------------------------------------------------------------
DECISION:
  - Strong bias is OK when user keeps correcting same pattern→word; not "lock-in"
    because user can retrain via slightly different swipe or repeated reverse corrections.
  - Decay applies to stale habits when user stops reinforcing.
  - fam/fan example: invisible M grow left, N may shift; user can still swipe N for "fan".
  - Word-level autocorrect override ONLY after repeated correction of BOTH similar
    swipe pattern AND key sequence to a specific replacement word.

SECTION C — ML / weights — UPDATED LIMITS
--------------------------------------------------------------------------------
DECISION:
  - Train on human corrections for keys, swipe patterns, and words.
  - Frequent Typo List promotion (density-based, tunable in testing):
      examples from user: 5+/day × 3 days; OR 20+/day; OR 15+/hour; OR 5+/15min
      for same wrong string (e.g. tye→the). Exact formula TBD on phone.
  - Protected words (the, and, of, …): cannot use naive "all 2–3 letter words immune"
      (breaks tye/the). Per-word allowlist + promotion logic instead.
  - Key shrink: max 20% (min 80% visual/hit baseline). Never grow beyond gap to
    neighbors; no overlap; invisible position shift: edge must not cross original
    key center; adjacent keys shrink/grow to meet halfway if both need space.
