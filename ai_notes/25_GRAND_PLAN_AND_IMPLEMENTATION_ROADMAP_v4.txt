A. PRODUCT IN ONE PAGE
================================================================================

Build a keyboard where:
  1. SWIPE BASE ("key-accuracy input") decodes finger path → letters → words using
     any installed standard dictionary (no proprietary swipe dictionary, no swipe lib).
     It is literal first; it may be wrong for bad spellers until trained.
  2. AUTOCORRECT & LEARNING SYSTEM runs alongside always: logs swipes, taps, and
     corrections even when autocorrect is OFF; quietly tunes swipe biases and key
     geometry; when ON, may change committed text for real spelling/grammar fixes.
  3. CONTEXT ENGINE (sub-feature of autocorrect) uses OSS grammar / n-gram data to
     ELIMINATE impossible next-word suggestions — never to invent or heavily pick words.
  4. COMMON TYPO LIST suppresses chronic mistake strings from pipeline output.
  5. All on-device; user can reset learning; no cloud AI.

Accuracy goal: movement → keys → dictionary, improved by user history — not commercial
keyboard-style whole-word guessing (food/find failure mode).

================================================================================
B. TWO-SYSTEM ARCHITECTURE
================================================================================

Think of two cooperating products inside one IME:

┌─────────────────────────────────────────────────────────────────────────────┐
│ SYSTEM 1 — SWIPE / TAP BASE ENGINE ("the keyboard")                         │
├─────────────────────────────────────────────────────────────────────────────┤
│ Job: Turn input into text using key-accuracy + dictionary prefix narrowing. │
│ Uses: Installed .dict (trie), coordinate map, per-segment key confidence.   │
│ Output: Commits top path match on swipe lift; runner-ups in suggestion strip. │
│                                                                             │
│ Always-on quiet self-improvement (even if autocorrect OFF):                 │
│   • Adjust swipe key biases / path weights from USER edits & strip picks    │
│   • Adjust invisible key size/position (within limits) from USER behavior   │
│   • Does NOT need autocorrect ON to learn from user fixing swipe output      │
│                                                                             │
│ User control: "Reset swipe learning" / "Reset key geometry" in settings.    │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ SYSTEM 2 — AUTOCORRECT & LEARNING ORCHESTRATOR                            │
├─────────────────────────────────────────────────────────────────────────────┤
│ Job: Intent recognition — fix spelling errors; apply chronic typo rules;    │
│      coordinate training signals to System 1.                               │
│                                                                             │
│ ALWAYS RUNNING (logging & quiet tuning authority):                          │
│   • Paragraph journal, WordSlots, heatmap DB, curve buckets                 │
│   • Records swipe patterns + outcomes + corrections regardless of AC toggle │
│   • May update System 1 weights quietly (same as user-driven training)      │
│                                                                             │
│ WHEN AUTOCORRECT OFF:                                                       │
│   • Must NOT change committed text in the text field                        │
│   • MAY still populate suggestion strip with proposed fixes (user optional) │
│   • Strip-only suggestions still train System 1 if user selects one         │
│                                                                             │
│ WHEN AUTOCORRECT ON:                                                        │
│   • May replace committed text for spelling/near-match fixes (spellng→      │
│     spelling); grammar-invalid word in sentence (rate→rare in context)       │
│   • Uses swipe path + keystrokes + key shifts + past corrections + context  │
│     eliminator — NOT blind LM word guessing                                 │
│   • If user accepts AC change → may accelerate training for that swipe path │
│   • If user reverts AC → train toward user's word, not AC suggestion        │
└─────────────────────────────────────────────────────────────────────────────┘

WHY THE SPLIT MATTERS
  • Base swipe will sometimes perform poorly alone (weak spellers, empty heatmap).
    That is expected early; System 2 fixes spelling/intent without replacing literal
    decode philosophy.
  • "Autocorrect" is not the same as "the keyboard guessing words." It is a layer
    that fixes errors and feeds learning back to System 1.
  • Turning autocorrect OFF still leaves a improving swipe experience over time.

C. LEARNING & TRAINING RULES
================================================================================

EVENT                              │ SYSTEM 1 (swipe/tap)      │ SYSTEM 2 (AC/log)
───────────────────────────────────┼───────────────────────────┼────────────────────
User completes swipe, keeps word   │ Reinforce path→word       │ Log accept
User picks runner-up in strip      │ Train path→that word      │ Log pick
User manually edits word in field  │ Train keys/path/geometry  │ Log correction
User tap-after-swipe at same slot  │ Gentle tap; match runnerUp│ Log mode switch
AC ON, replaces word, user keeps   │ Train path toward AC word │ Log AC accept
AC ON, user reverts to own word    │ Train toward USER word    │ Penalize AC pair
AC OFF                             │ Learn from edits only     │ No text mutation
Reset learning (settings)          │ Clear biases/geometry     │ Clear typo weights*
                                   │                           │ *Typo list optional separate reset

AUTOCORRECT FIX TYPES (when ON)
  T1  Near-match spelling: spellng / spellong → spelling (dictionary edit distance)
  T2  Valid isolated word: fleeting alone → no change
  T3  Grammar-invalid token in sentence: "this is so rate!" → rare (context engine)
  T4  Chronic typo: tye → the (Common Typo List + history)
  T5  Swipe+history intent: wrong path word corrected often → bias path (System 1)

Training priority: user actions > accepted AC > quiet inference from repeated patterns.

================================================================================
D. CONTEXT / GRAMMAR ENGINE (sub-feature of System 2)
================================================================================

VERDICT: Good fit — with strict role = ELIMINATION ONLY.

PURPOSE
  Remove suggestion candidates that are grammatically or usage-impossible in slot,
  not to choose the winning word. Swipe engine + user heatmap still pick among survivors.

SOURCES (must be FLOSS, on-device, bundled or downloadable)
  A) Open-source English n-gram / grammar constraint library (e.g. small n-gram
     tables, rule-based grammar checker lexicon — exact package TBD in implement step)
  B) User's own bigram/trigram stats from typed history (local, privacy-safe)

BEHAVIOR EXAMPLES
  After "OH MY" → eliminate "eating" as next-word suggestion (implausible continuation).
  After "you're going to" → eliminate "sponge", "chocolate" as next slot candidates.
  "this is so rate!" → flag "rate" as invalid in structure; allow "rare" as fix candidate because it is a highly similar swipe pattern and creates valid sentence structure.

WEIGHT RULE (locked)
  • Context/grammar may DISQUALIFY or heavily down-rank candidates (effectively -∞).
  • Context/grammar must NOT outweigh swipe path + user correction history for which
    key sequence was intended.
  • Think: grammar is a filter on the suggestion set, not the primary scorer.

OVERHEAD
  • Keep lexicon/n-grams lean (tens of MB max, mmap).
  • Run filter after candidate list built, before strip display / AC commit.
  • If perf issues: disable on low-RAM devices via settings.

SETTINGS
  Under Heatmap Smart Keyboard:
  [Toggle] Grammar context filter (default ON when bundled data present)

E. OTHER FEATURES (SHORT)
================================================================================

COMMON TYPO LIST — System 2; pipeline ban; tap-twice escape; density promotion (tune on device);
  function-word guard (the, and, …) not length immunity.

SETTINGS HUB — "Heatmap Smart Keyboard": learning master, sensitive fields (default off),
  paragraph slider 500–3000 (snap 100), AC toggle, reset learning, typo list, diagnostics export,
  grammar filter toggle, geometry toggles.

KEY GEOMETRY — System 1 quiet adjust; 80% min size; grow to gap only; no overlap;
  tap/swipe separate; reset buttons.

DIAGNOSTICS — Permanent: session record, JSON export, heatmap overlay, metrics.
