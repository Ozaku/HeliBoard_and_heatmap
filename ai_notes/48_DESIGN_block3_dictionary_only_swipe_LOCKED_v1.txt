DECODE PIPELINE (target architecture)
  1. Tap/swipe gate (13a) — min axis span per start key; micro-drag = tap
  2. Anchor first letter from stroke START key region (strict)
  3. Walk stroke geometry:
     - Sharp angle / tight curve at point → letter hypothesis at that beat
     - Direction change (zig-zag) → key at each corner/point
     - Large wide-angled gentle arc → gap BETWEEN keys (no letter); not a miss
     - Tight loop with multiple tight curls → multiple letters in that region
  4. Build prefix / key sequence from beats (not from glide LM guess)
  5. Narrow candidate words:
     - Trie prefix walk on TAP dictionaries only (BinaryDictionary isInDictionary / prefix)
     - Boost / bias from user correction weights (46_) and path signatures (step 10)
  6. Score survivors with 44_ letter weights (first strict, middle loose, last moderate)
  7. On lift: commit top match; runner-ups in strip for user correction training

LEARNING LOOP (ongoing, no corpus typos)
  Each swipe session + correction updates:
    - letter_confusion, word_correction (step 12)
    - path_bucket + alignment when SWIPE commits exist (step 10)
  Geometry decode gets better per user over sessions; NOT from crowd typo stats.

GEOMETRY RULES SUMMARY
  | Stroke feature              | Interpretation              |
  |-----------------------------|-----------------------------|
  | Sharp angle / tight curve   | Letter at/near that point   |
  | Zig-zag corners             | Letter at each corner       |
  | Wide gentle arc             | Between keys (skip letter)  |
  | Tight multi-curl loop       | Multiple letters in loop    |
  | Start key region            | First letter (hard anchor)  |
  | End region                  | Soft anchor (44_)           |
